# Plan: Rewrite MacDown as a Tauri app

## Context

MacDown is a ~10-year-old Objective-C/Cocoa Markdown editor. The current toolchain is brittle: it requires Xcode, a working Ruby + CocoaPods (which broke against macOS system Ruby and again against modern Ruby — see `memory/macdown_gemfile_travis.md`), and an old Sparkle/MASPreferences/PAPreferences stack pinned to 10.8/10.14 era APIs. Building it cleanly on a fresh 2026 Mac took multiple Gemfile patches.

The fortunate observation is that **most of MacDown's interesting rendering already runs as JavaScript inside a WebView**: Prism, MathJax, mermaid, and viz.js are all bundled as web assets and executed in the preview pane. The only major native pieces are (1) the syntax-highlighted text editor, (2) the C-based Hoedown Markdown→HTML step, and (3) macOS niceties like Sparkle, Services, and Touch Bar.

Goal: replace MacDown with a **Tauri** application — a Rust shell hosting a system WebView (WKWebView on macOS, WebView2 on Windows, WebKitGTK on Linux). Outcome: one TypeScript+Rust codebase, no Ruby, no Xcode required for development, ~10MB bundle, cross-platform from day one, native macOS feel via WKWebView.

This plan does not modify the existing repo. It describes a new project that reaches feature parity for macOS first, then expands to Windows/Linux.

## Target architecture

- **Rust shell (Tauri)** — windowing, native menus, file I/O, dialogs, settings store, auto-updater, sidecar CLI, OS-specific integrations.
- **TypeScript frontend** — editor pane, preview pane, preferences UI, all rendering.
  - Editor: **CodeMirror 6** with Markdown mode + custom decorations to match peg-markdown-highlight's behavior.
  - Markdown→HTML: **markdown-it** with plugins matching the Hoedown extension set MacDown enables today.
  - Code highlighting in preview: **Prism** (port verbatim).
  - Math: **MathJax** (port verbatim) or **KaTeX** (lighter, faster) — pick during step 4.
  - Diagrams: **mermaid** + **viz.js** (port verbatim).
  - Themes: existing CSS theme files copied as-is.
  - i18n: **i18next** with JSON files generated from MacDown's `.strings`.

## Feature mapping (current → new)

| MacDown feature | Tauri replacement |
|---|---|
| Hoedown | markdown-it + plugins (gfm, footnotes, deflist, task-lists, sub/sup, emoji, autolink, ins/mark) |
| peg-markdown-highlight (editor) | CodeMirror 6 markdown mode + custom decorations |
| Prism / MathJax / mermaid / viz.js | unchanged — copy from `MacDown/Code/Resources` |
| MASPreferences/PAPreferences | Settings modal in webview, persisted via `tauri-plugin-store` |
| Sparkle | Built-in `tauri-plugin-updater` (signed-update support included) |
| handlebars-objc (HTML export) | Handlebars JS |
| `macdown-cmd` | Tauri sidecar Rust binary, IPC to the running app |
| 26 .lproj `.strings` files | i18next JSON, generated by a one-shot script |
| AppleScript `.sdef` | **Drop** unless explicitly requested later |
| Touch Bar | **Drop** (Apple has deprecated it) |
| Services menu | **Defer** to phase 2 (macOS-specific Tauri menu work) |

## Files in the existing repo that must be referenced (read-only) during the port

- `MacDown/Code/Document/MPRenderer.m` — authoritative list of Hoedown extensions and HTML render flags. The new markdown-it config must enable the same set or behavior will diverge silently.
- `MacDown/Code/Document/MPDocument.m` — debounce timing, file-change observation patterns, and editor↔preview scroll-sync logic.
- `MacDown/Code/Resources/` — preview HTML template, Prism/MathJax/mermaid initialization, CSS themes (`Themes/`), syntax highlight styles. Most of this can be copied byte-for-byte.
- `MacDown/Code/Preferences/` — every preference key + default value. Mirror these in the new settings store so existing user expectations carry over.
- `MacDown/Localization/*.lproj/*.strings` — translation source.

## Phased execution

Each phase is independently shippable internally; do not start phase N+1 until N is demonstrably correct.

1. **Skeleton.** `pnpm create tauri-app` (TypeScript template). Get a macOS window with an empty editor pane and an empty preview pane. Set up the build pipeline and CI matrix (macOS only at first).
2. **Editor + file lifecycle.** Wire CodeMirror 6, implement Open/Save/Save As via Tauri's `dialog` and `fs` APIs, watch for external changes. Debounced re-render.
3. **Markdown rendering.** Build the markdown-it pipeline matching `MPRenderer.m`'s extension set. Pull `MacDown/Code/Resources/Themes/*.css` into the new repo. Validate by rendering a corpus of `.md` files in both apps and diffing the HTML.
4. **Code highlight + math + diagrams.** Drop Prism, MathJax (or evaluate KaTeX), mermaid, viz.js into the preview HTML. Copy their initialization from MacDown's `template.html`.
5. **Preferences.** Build the settings modal. Mirror every key from `MacDown/Code/Preferences/`. Persist via `tauri-plugin-store`.
6. **Localization.** Write a converter script: `.strings` (and `.stringsdict`) → i18next JSON. Keep keys identical so existing Transifex contributions still apply.
7. **HTML/PDF export.** HTML export via Handlebars JS using the existing template. PDF via the WebView's print-to-PDF.
8. **Updater.** Configure `tauri-plugin-updater`. Decide whether to migrate the existing Sparkle appcast feed or run a parallel new feed.
9. **CLI helper.** Port `macdown-cmd` as a Tauri sidecar Rust binary that connects via local IPC.
10. **Windows + Linux smoke.** Build, open a corpus of docs, fix the obvious WebView2/WebKitGTK quirks.
11. **Signing & distribution.** macOS notarization still needs Apple's `codesign`+`notarytool` (Command Line Tools, not Xcode.app). Document the release pipeline.

## Risks worth naming up front

- **Editor parity is the hardest part.** Users will notice if Markdown autocomplete, table editing, smart quotes, or list-continuation behavior differs. Budget real time for CodeMirror 6 customization.
- **markdown-it ≠ Hoedown.** Edge-case Markdown produces different HTML. The phase 3 corpus diff is non-negotiable.
- **macOS-native muscle memory.** Services, AppleScript, drag-receive from Finder thumbnails, and the macOS spell-checker integration are easy to underestimate. Decide explicitly which to keep.
- **Plugin API.** The current Obj-C `MPPlugInController` plugin system would need a new design (probably JS-based plugins). If anyone actually uses MacDown plugins, this is a breaking change worth flagging in release notes.
- **Bundle distribution.** Tauri makes `.app`/`.dmg`/`.msi`/`.AppImage`/`.deb` straightforward, but the first signed+notarized macOS build always involves friction with Apple Developer ID. Allocate a day for it.

## Verification

- **Render parity:** harness that loads a corpus of `.md` files (the MacDown test fixtures + a curated set of GFM edge cases), renders in both apps, and diffs the HTML output. Target: <5% non-cosmetic diff after phase 3.
- **Theme parity:** screenshot-compare each MacDown CSS theme against the Tauri build. Pixel-similar (font rendering will differ slightly between WKWebView and Cocoa text system).
- **Locale check:** boot the app under each of the 26 locales; spot-check that menus and the settings modal are translated.
- **Cross-platform smoke:** open the same corpus on Windows and Linux; confirm no JavaScript errors and the preview renders.
- **Updater dry run:** stage a fake newer release; confirm the in-app updater detects, downloads, and applies it on macOS, Windows, and Linux.

## Effort estimate

Solo developer, full-time: roughly **4–8 weeks** to reach macOS parity, **+2–4 weeks** for cross-platform polish and distribution. Most of that time is spent on the editor (phase 2) and rendering parity (phase 3); the rest is plumbing.
